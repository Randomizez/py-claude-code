# Context

This document records the current Anthropic Messages request and context shape used by `pyccode`.

It follows the style of `pycodex/docs/CONTEXT.md`, but the semantic baseline here is repository A rather than the OpenAI Responses API.

This document is not a raw packet dump and not an implementation diary. It is a context contract.

Conventions:

- `A ...` means the baseline behavior derived from or captured from repository A.
- `Our ... Diff` means only the behavior that differs in `pyccode`.
- `status = "captured"` means verified by a real local capture.
- `status = "derived"` means inferred from A source and current captures.
- `status = "implemented"` means directly confirmed in current `pyccode` code.

Notation:

- examples use Python `dict`, `list`, and f-string style
- dynamic values use semantic placeholders such as:
  - `"{cwd()}"`
  - `"{date(YYYY-MM-DD)}"`
  - `"{local timezone name}"`
  - `"{ANTHROPIC_MODEL or --model}"`

## 0. `system`, `messages`, and `tools` Are Separate Layers

Anthropic Messages differs structurally from the OpenAI Responses model used in `pycodex`.

```python
AnthropicMessagesRequest = {
    "model": "{model slug}",
    "system": [*system_blocks],
    "messages": [*conversation_messages],
    "tools": [*tool_schemas],
    "max_tokens": 32000,
}
```

Important points:

- `system` is a top-level field, not a user message
- `messages` carries the turn history
- `tools` is also top-level
- A separates prompt, system context, and user-context shaping more aggressively than `pyccode`

### 1.1 Baseline from A

At a high level, the first-turn request in A can be summarized as:

```python
ABaselineFirstTurnRequest = {
    "status": "derived_and_partly_captured",
    "body": {
        "model": "{runtime main-loop model}",
        "system": [
            {"type": "text", "text": "{system block 1}"},
            {"type": "text", "text": "{system block 2}"},
            {"type": "text", "text": "{dynamic context block}"},
        ],
        "messages": [
            *normalized_messages,
        ],
        "tools": [
            *filtered_tool_schemas,
        ],
        "metadata": {
            "user_id": "{json string with device_id, account_uuid, session_id}",
        },
        "thinking": (
            None
            | {"type": "adaptive"}
            | {"type": "enabled", "budget_tokens": "{<= max_tokens - 1}"}
        ),
        "context_management": (
            None
            | {"edits": [*context_edits]}
        ),
        "tool_choice": (
            None
            | {"type": "auto"}
            | {"type": "tool", "name": "{tool name}"}
        ),
        "max_tokens": "{runtime max output tokens}",
        "stream": True,
    },
}
```

Interpretation:

- A layers system prompt, system context, and user context instead of flattening everything into one string.
- A normalizes outbound `messages` before sending them.
- A constructs tool schemas dynamically from the active runtime tool set.
- A consistently carries session-oriented metadata and higher-level request controls.

### 1.6 Our Current Diff Relative to A

The main remaining first-turn gaps are:

- `system` is still somewhat flatter than in A
- outbound `messages` still undergo less normalization than in A
- several runtime-controlled fields remain simplified
- the default visible tool surface is schema-aligned first; execution alignment is still partial
- some differences are inherently dynamic, especially session identifiers

## 2. Tool Follow-Up Request

### 2.1 Baseline from A

A keeps a strict pairing between assistant `tool_use` blocks and user `tool_result` blocks.

```python
AToolFollowUp = {
    "status": "derived",
    "messages": [
        *history_before_tool_use,
        {
            "role": "assistant",
            "content": [
                {"type": "tool_use", "id": "...", "name": "...", "input": {...}},
            ],
        },
        {
            "role": "user",
            "content": [
                {"type": "tool_result", "tool_use_id": "...", "content": [...]},
            ],
        },
    ],
}
```

### 2.2 Current `pyccode` Implementation

```python
PyccodeToolFollowUp = {
    "status": "implemented",
    "behavior": [
        "store the assistant reply in history first",
        "execute the full tool batch",
        "merge the tool outputs into one user message for the next turn",
        "preserve thinking blocks in memory history while omitting them from the default outbound request",
    ],
}
```

### 2.3 Remaining Diff

- A performs more message normalization around tool follow-up turns
- A also has richer attachment, hook, compaction, and memory-related message types
- thinking retention rules are stricter in A
- tool-result wrapping is only partially aligned today

## 4. Recommended Convergence Order

The most sensible order for future context work is:

1. keep the current real request dumps stable in the generated debug artifacts
2. continue tightening the mapping from A's prompt and context layers into `pyccode/context.py`
3. move message normalization and tool-result pairing closer to A
4. decide only later whether to add wider context sources such as memory, compaction, and hooks
